1. WHAT THE PRODUCT IS
--------------------------------------------------------------------------------

Name:      SGRH Lab AI Assistant
Owner:     Sir Ganga Ram Hospital — Information Technology Department
Audience:  Hospital IT leadership and lab operations management (non-engineers),
           plus the engineering team. The diagram is shown in slide decks and
           printed in documents.

What it does: a web dashboard where a lab manager asks a question in plain
English — "Revenue share of OPD vs IPD patients", "Average turnaround time by
test", "Cost per sample this quarter" — and gets back a one-sentence finding
plus charts and tables. It is scoped to Biochemistry Lab operations across four
metric domains: COST, REVENUE, SERVICE (test volumes) and TAT (turnaround time).

Scale of the live data: 11 tables, 5,217,019 rows in Azure PostgreSQL. The two
largest tables hold ~1.5M and ~1.02M rows.


--------------------------------------------------------------------------------
2. THE ONE IDEA THE DIAGRAM MUST COMMUNICATE
--------------------------------------------------------------------------------

Every number shown to a user is computed by the DATABASE across COMPLETE tables
— never inferred by the language model from a sample of rows.

This is not a stylistic claim; it is the system's defining design decision, and
it exists because the opposite was tried and failed. An earlier version loaded
500 rows per table into memory and let the model aggregate those. Asked for the
OPD vs IPD revenue split, it answered "OPD 61.4%" with total confidence. The
truth, over all 1,021,711 rows, is the reverse: IPD 59.0%, OPD 41.0%. It had
seen 0.05% of the data and stated the wrong direction as fact.

So stage 2 ("Aggregate") is the centre of gravity of this diagram. If a viewer
takes away exactly one thing, it should be that exact figures come from SQL over
whole tables. Consider giving that stage subtle visual emphasis — but do not
turn it into a garish highlight; the tone is clinical, not promotional.

6. WHAT TO IMPROVE
--------------------------------------------------------------------------------

The current diagram is correct and readable but visually plain. Priorities, in
order — the first one matters most:

  1. BIGGER TEXT, TIGHTER BOXES. This is the single most important change.
     The type is too small and the boxes are too loose, so the diagram reads as
     mostly empty space with tiny labels floating in it. Fix both together:

       - Increase every type size substantially. The body text should be
         comfortably readable when the diagram is projected or printed at A4,
         not merely legible when zoomed in. Roughly a 25-40% increase across
         the scale is the right order of magnitude. Scale headings, card
         titles, body and captions together so the hierarchy is preserved.

       - Cut the padding between text and the box that contains it. Cards
         currently have far more internal margin than their content needs,
         and the four engine stage cards have obvious dead space below their
         last line of text. Text should sit closer to its border.

       - Cut the gaps between cards, and between cards and the band around
         them, for the same reason.

     The goal: the same information, in the same layout, at a noticeably
     larger text size, achieved by reclaiming wasted padding rather than by
     enlarging the canvas. Boxes should look sized to their content — snug,
     not cramped and not airy. Keep enough padding that nothing touches a
     border, but no more than that.

  2. Hierarchy. Four zones (sources / engine / outputs / model) currently carry
     near-equal visual weight. The engine is the subject and should read first;
     the model cloud is a dependency, not a peer.

  3. The stage sequence. Four numbered cards with small arrows between them
     under-sells that this is a pipeline. Consider a stronger flow treatment —
     but keep the numbering, since it is how people will refer to the stages.

  4. Iconography. Cards are currently identified only by a coloured rule. A
     restrained, consistent icon set (line icons, single weight, no fills)
     would help scanning. Icons must be literal and unambiguous: a database
     cylinder, a document, a chart, a table, a chat bubble. If good icons
     cannot be made consistent, keep the coloured rules — inconsistent icons
     are worse than none.

  5. Depth and separation. Everything sits on one flat plane with hairline
     borders. Very subtle elevation on cards, or a slight tonal step between
     the page, the bands and the cards, would improve legibility.

  6. Connectors. Arrows are plain lines with triangular heads. They deserve
     more considered routing, weight and label placement — labels currently
     float near the lines rather than being clearly attached to them.

  7. Consistency. Internal padding is currently uneven between card types.
     Establish one spacing scale and apply it everywhere, so equivalent
     elements are spaced identically.

  8. Print and projection. It must survive being shrunk into a slide, printed
     in greyscale, and viewed on a projector. Do not rely on colour alone to
     distinguish anything — the four stages must still be tellable apart in
     black and white via their numbers and labels.

--------------------------------------------------------------------------------
8. USEFUL BACKGROUND (context, not content for the diagram)
--------------------------------------------------------------------------------

  - Frontend: React 18, TypeScript, Vite, Tailwind, Plotly charts, light/dark
    theming, unit-tested and CI-gated.
  - Backend: a single FastAPI service. It holds a loaded dataset in memory as a
    session, exposes REST endpoints to the web app, and runs an MCP server in a
    background thread so external AI agents can use the same tools.
  - The model is reached over HTTPS through Azure's OpenAI-compatible endpoint.
    It never touches the database directly — it receives pre-computed exact
    figures and returns prose plus chart definitions. This separation is the
    reason the numbers are trustworthy, and is worth making visually evident.
  - "OPD" = outpatient, "IPD" = inpatient, "TAT" = turnaround time. These are
    the domain terms the users think in.
